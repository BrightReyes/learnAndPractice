# SECTION 1: HR / INTRODUCTION QUESTIONS

> These are almost always asked first. Your answers set the tone for the entire interview. Philippine companies especially value attitude, willingness to learn, and cultural fit.

---

### Q1: "Tell me about yourself."

**Ideal Answer:**

> "Good day, sir/ma'am. I'm [Name], a [Year Level] BSIT student at [University]. I've been passionate about web development since [Year/Semester], and I've built several projects using HTML, CSS, JavaScript, PHP, and ReactJS. One project I'm most proud of is [Project Name] — it's a [brief description] where I implemented [key feature]. **Why this answer is strong:**
- Follows the **Present → Past → Future** structure
- Mentions specific technologies matching the job requirements
- Shows self-driven learning (studied NestJS, Prisma, PostgreSQL on your own)
- Ends by connecting your goals to the company

**Common mistakes:**
- Reciting your entire resume chronologically
- Starting with "Uhm, so basically…"
- Being too vague ("I like coding")
- Going longer than 90 seconds

**Follow-up questions:**
- "What made you start learning web development?"
- "Tell me more about that project."

# SECTION 2: HTML

> HTML is almost always tested as a foundational topic. Interviewers want to see that you understand semantics, accessibility, and forms — not just how to write tags.

---

### Q1: What is the difference between block-level and inline elements? Give examples.

**Ideal Answer:**

> Block-level elements take up the full width available and always start on a new line. Examples: `<div>`, `<p>`, `<h1>`, `<section>`, `<form>`. Inline elements only take up as much width as necessary and don't start a new line. Examples: `<span>`, `<a>`, `<strong>`, `<img>`, `<input>`.

**Why it's strong:** Shows understanding of the fundamental layout behavior difference, with correct examples.

**Common mistake:** Saying `<img>` is a block element (it's actually inline-replaced).

**Follow-up:** "Can you change a block element to inline?" → Yes, with `display: inline` in CSS.

---

### Q2: What is semantic HTML and why does it matter?

**Ideal Answer:**

> Semantic HTML means using elements that clearly describe their meaning — like `<header>`, `<nav>`, `<main>`, `<article>`, `<section>`, `<footer>` instead of generic `<div>` tags everywhere. It matters for three reasons: (1) **SEO** — search engines understand page structure better; (2) **Accessibility** — screen readers can navigate the page properly for visually impaired users; (3) **Maintainability** — other developers understand the code faster.

**Common mistake:** Only mentioning SEO and not accessibility.

---

### Q3: What is the difference between `<section>`, `<article>`, and `<div>`?

**Ideal Answer:**

> `<article>` is for self-contained content that makes sense on its own — like a blog post, news article, or product card. `<section>` is for thematic grouping of related content, usually with a heading — like "About Us" or "Features" sections on a landing page. `<div>` is a generic container with no semantic meaning — use it only when no semantic element fits, usually for styling/layout purposes.

---

### Q4: What is the purpose of the `DOCTYPE` declaration?

**Ideal Answer:**

> `<!DOCTYPE html>` tells the browser to render the page in **standards mode** rather than quirks mode. Without it, browsers might use quirks mode which emulates legacy behavior and can cause inconsistent rendering across browsers. It must be the very first line in an HTML document.

---

### Q5: Explain the difference between `GET` and `POST` methods in forms.

**Ideal Answer:**

> `GET` appends form data to the URL as query parameters — it's visible in the address bar, can be bookmarked, has a length limit (~2048 characters), and is used for retrieving data (search forms, filters). `POST` sends data in the request body — it's not visible in the URL, has no practical size limit, and is used for submitting data that changes state (login, registration, creating records). You should never use `GET` for sensitive data like passwords.

**Follow-up:** "When would you use GET over POST?" → For search forms, pagination, filtering — anything where you want the user to be able to bookmark or share the URL.

---

### Q6: What are `data-*` attributes?

**Ideal Answer:**

> `data-*` attributes allow you to store custom data on any HTML element. The data is accessible in JavaScript via `element.dataset`. For example, `<button data-user-id="42">Delete</button>` — in JavaScript you access it with `button.dataset.userId`. They're useful for passing data from HTML to JavaScript without using hidden inputs or global variables.

```html
<tr data-row-id="5" data-status="active">
  <td>John</td>
</tr>
<script>
  const row = document.querySelector('tr');
  console.log(row.dataset.rowId);   // "5"
  console.log(row.dataset.status);  // "active"
</script>
```

---

### Q7: What is the difference between `localStorage`, `sessionStorage`, and cookies?

**Ideal Answer:**

> `localStorage` persists data with no expiration — it stays even after the browser is closed. It can store about 5-10MB and is only accessible via JavaScript. `sessionStorage` stores data for the duration of the browser session — it's cleared when the tab is closed. Cookies are smaller (about 4KB), can be set to expire, and are sent automatically with every HTTP request to the server — making them useful for authentication tokens.

| Feature | localStorage | sessionStorage | Cookies |
|---------|-------------|----------------|---------|
| Capacity | ~5-10MB | ~5MB | ~4KB |
| Expires | Never | On tab close | Configurable |
| Sent to server | No | No | Yes, every request |
| Accessible | JavaScript only | JavaScript only | JS & Server |

---

### Q8: What is the difference between `<script>`, `<script async>`, and `<script defer>`?

**Ideal Answer:**

> `<script>` — Blocks HTML parsing until the script is downloaded and executed. `<script async>` — Downloads the script in parallel but executes immediately when ready, potentially before HTML is fully parsed. `<script defer>` — Downloads in parallel and waits to execute until HTML parsing is complete. Use `defer` for scripts that need the DOM to be ready; use `async` for independent scripts like analytics.

---

### Q9: How do you make a website accessible?

**Ideal Answer:**

> Key practices: (1) Use semantic HTML elements. (2) Add `alt` text to all images. (3) Ensure proper heading hierarchy (`h1` → `h2` → `h3`, don't skip). (4) Use `<label>` elements linked to form inputs with `for` attributes. (5) Ensure keyboard navigation works — all interactive elements should be focusable. (6) Use ARIA attributes when needed (`aria-label`, `aria-describedby`, `role`). (7) Maintain sufficient color contrast. (8) Add skip navigation links.

---

### Q10: What is the difference between `<strong>` and `<b>`, and between `<em>` and `<i>`?

**Ideal Answer:**

> `<strong>` has semantic meaning — it indicates important text, and screen readers will emphasize it. `<b>` just makes text bold visually with no semantic meaning. Similarly, `<em>` means emphasized text (screen readers change tone), while `<i>` is purely visual italics. Always prefer `<strong>` and `<em>` for accessibility.

---

### Q11: What is the `<meta viewport>` tag and why is it important?

**Ideal Answer:**

> `<meta name="viewport" content="width=device-width, initial-scale=1.0">` tells mobile browsers to set the viewport width to the device width and the initial zoom to 100%. Without it, mobile browsers render the page at a desktop width (typically 980px) and scale it down, making text tiny and unusable. It's essential for responsive design.

---

### Q12: What are void (self-closing) elements?

**Ideal Answer:**

> Void elements are HTML elements that cannot have child content and don't need a closing tag. Examples: `<img>`, `<input>`, `<br>`, `<hr>`, `<meta>`, `<link>`. In HTML5, the trailing slash is optional (`<br>` and `<br />` are both valid), but in XHTML the slash is required.

---

### Q13: Explain the difference between `<link>` and `<a>`.

**Ideal Answer:**

> `<link>` is used in the `<head>` section to define relationships between the document and external resources — like stylesheets (`<link rel="stylesheet" href="style.css">`), favicons, and preloaded fonts. `<a>` (anchor) is used in the `<body>` to create clickable hyperlinks that navigate users to other pages or sections.

---

### Q14: What is the purpose of the `<figure>` and `<figcaption>` elements?

**Ideal Answer:**

> `<figure>` wraps self-contained content like images, diagrams, or code snippets that are referenced in the main text. `<figcaption>` provides a caption for the figure. This is semantically better than just putting an image and a paragraph together, because it explicitly associates the caption with the content.

```html
<figure>
  <img src="dashboard.png" alt="Dashboard screenshot showing analytics data">
  <figcaption>Figure 1: Admin dashboard with real-time analytics</figcaption>
</figure>
```

---

### Q15: What is an iframe? When should and shouldn't you use it?

**Ideal Answer:**

> An `<iframe>` embeds another HTML document inside the current page — like embedding a YouTube video, Google Map, or third-party widget. You should use it for embedding trusted third-party content. You should NOT use it for embedding pages from your own site (use components or includes instead), and you need to be careful about security — always set the `sandbox` attribute to restrict what the embedded content can do.

---

### Q16: What happens when you type a URL in the browser and press Enter? (HTML perspective)

**Ideal Answer:**

> The browser sends an HTTP GET request to the server. The server responds with an HTML document. The browser parses the HTML top-to-bottom, building the DOM tree. When it encounters `<link>` tags, it fetches CSS and builds the CSSOM. When it encounters `<script>` tags, it downloads and executes JavaScript (which may block parsing unless `async` or `defer`). Once DOM and CSSOM are ready, the browser combines them into a render tree, calculates layout, and paints pixels on screen.

---

### Q17: Coding Question — Create a responsive two-column layout using only HTML.

**Ideal Answer:**

```html
<main>
  <section>
    <h2>Main Content</h2>
    <article>
      <h3>Article Title</h3>
      <p>Article content goes here...</p>
    </article>
  </section>
  <aside>
    <h2>Sidebar</h2>
    <nav>
      <ul>
        <li><a href="#category1">Category 1</a></li>
        <li><a href="#category2">Category 2</a></li>
      </ul>
    </nav>
  </aside>
</main>
```

> Note: HTML alone doesn't create columns — you'd use CSS for that. But the semantic structure using `<main>`, `<section>`, `<aside>`, and `<nav>` is what interviewers want to see.

---

### Q18: Debugging — Why is this form not submitting properly?

```html
<form action="/submit">
  <input type="text" placeholder="Name">
  <input type="email" placeholder="Email">
  <div onclick="submitForm()">Submit</div>
</form>
```

**Ideal Answer:**

> Several issues: (1) The `<div>` is not a `<button type="submit">`, so clicking it won't trigger form submission natively. (2) No `method` attribute — it defaults to GET, which may not be desired. (3) Missing `name` attributes on inputs — without `name`, the server won't receive the data. (4) Missing `<label>` elements for accessibility. (5) No `required` validation attributes.

**Fixed version:**
```html
<form action="/submit" method="POST">
  <label for="name">Name:</label>
  <input type="text" id="name" name="name" required>
  
  <label for="email">Email:</label>
  <input type="email" id="email" name="email" required>
  
  <button type="submit">Submit</button>
</form>
```

# SECTION 3: CSS

> CSS questions in Philippine internship interviews tend to focus on layout (Flexbox, Grid), responsive design, the box model, and specificity. Be ready to write CSS on the spot.

---

### Q1: Explain the CSS Box Model.

**Ideal Answer:**

> Every HTML element is a rectangular box with four layers from inside out: **Content** (the actual text/image), **Padding** (space between content and border), **Border** (the edge of the box), and **Margin** (space between this box and other boxes). By default, `width` only sets the content width, so total width = content + padding + border. Using `box-sizing: border-box` makes `width` include padding and border, which is much more predictable.

```css
/* Without border-box: total width = 300 + 20 + 20 + 2 + 2 = 344px */
.card { width: 300px; padding: 20px; border: 2px solid black; }

/* With border-box: total width = 300px (padding and border included) */
.card { width: 300px; padding: 20px; border: 2px solid black; box-sizing: border-box; }
```

**Follow-up:** "Why do most CSS resets include `box-sizing: border-box`?" → Because it makes sizing elements intuitive — the width you set is the width you get.

---

### Q2: What is the difference between `margin` and `padding`?

**Ideal Answer:**

> `padding` is the space **inside** the element — between the content and the border. It pushes the content inward and the background color fills it. `margin` is the space **outside** the element — between the border and neighboring elements. Margins can also collapse (two adjacent vertical margins combine into one), while padding never collapses.

---

### Q3: Explain CSS specificity and how conflicts are resolved.

**Ideal Answer:**

> When multiple CSS rules target the same element, specificity determines which wins. The hierarchy from highest to lowest: (1) `!important` (avoid using), (2) Inline styles (`style=""`) — weight 1000, (3) ID selectors (`#id`) — weight 100, (4) Class selectors (`.class`), attribute selectors, pseudo-classes — weight 10, (5) Element selectors (`div`, `p`) — weight 1. If specificity is equal, the rule that appears last wins.

```css
p { color: blue; }                  /* Specificity: 0-0-1 */
.text { color: red; }              /* Specificity: 0-1-0 → wins over p */
#intro { color: green; }           /* Specificity: 1-0-0 → wins over .text */
p.text#intro { color: purple; }    /* Specificity: 1-1-1 → wins over all above */
```

---

### Q4: What is Flexbox and when would you use it?

**Ideal Answer:**

> Flexbox is a one-dimensional layout system that lets you arrange items in a row or column with flexible sizing and alignment. It's ideal for: navigation bars, centering content, distributing space evenly between items, card layouts in a single row, and aligning elements vertically and horizontally. The parent becomes a flex container with `display: flex`, and children become flex items.

```css
.container {
  display: flex;
  justify-content: space-between; /* Distributes items along main axis */
  align-items: center;            /* Centers items along cross axis */
  gap: 16px;                      /* Space between items */
}

.item {
  flex: 1;  /* All items take equal width */
}
```

**Follow-up:** "What's the difference between `justify-content` and `align-items`?" → `justify-content` controls the **main axis** (horizontal by default), `align-items` controls the **cross axis** (vertical by default).

---

### Q5: What is CSS Grid and how does it differ from Flexbox?

**Ideal Answer:**

> CSS Grid is a two-dimensional layout system that handles both rows AND columns simultaneously. Flexbox is one-dimensional — it handles either a row OR a column at a time. Use Grid for overall page layouts (header, sidebar, main, footer) and complex two-dimensional layouts. Use Flexbox for simpler one-direction alignments like navbars or card rows.

```css
.page-layout {
  display: grid;
  grid-template-columns: 250px 1fr;        /* Sidebar + Main */
  grid-template-rows: 60px 1fr 40px;       /* Header + Content + Footer */
  grid-template-areas:
    "header header"
    "sidebar main"
    "footer footer";
  min-height: 100vh;
}

.header  { grid-area: header; }
.sidebar { grid-area: sidebar; }
.main    { grid-area: main; }
.footer  { grid-area: footer; }
```

---

### Q6: How do you center a `<div>` both horizontally and vertically?

**Ideal Answer:**

> The modern approach uses Flexbox:
```css
.parent {
  display: flex;
  justify-content: center;
  align-items: center;
  height: 100vh;
}
```
> Alternative with Grid (even shorter):
```css
.parent {
  display: grid;
  place-items: center;
  height: 100vh;
}
```

**Common mistake:** Using `margin: auto` alone — that only centers horizontally, not vertically (unless the parent is a flex or grid container).

---

### Q7: What are CSS media queries? Write one.

**Ideal Answer:**

> Media queries apply CSS rules based on the device's characteristics — most commonly screen width. They're the foundation of responsive design. Mobile-first approach means default styles are for mobile, and you add `min-width` queries for larger screens.

```css
/* Mobile-first (default styles for small screens) */
.container { padding: 16px; }

/* Tablet (768px and up) */
@media (min-width: 768px) {
  .container { padding: 24px; max-width: 720px; margin: 0 auto; }
}

/* Desktop (1024px and up) */
@media (min-width: 1024px) {
  .container { padding: 32px; max-width: 1200px; }
}
```

---

### Q8: What is the difference between `em`, `rem`, `px`, `%`, and `vh`/`vw`?

**Ideal Answer:**

| Unit | Relative To | Best Used For |
|------|------------|---------------|
| `px` | Absolute (fixed) | Borders, shadows, small exact values |
| `em` | Parent element's font-size | Component-level spacing that scales with text |
| `rem` | Root (`<html>`) font-size | Global spacing, font sizes (most predictable) |
| `%` | Parent element's dimension | Fluid widths, responsive layouts |
| `vh`/`vw` | Viewport height/width | Full-screen sections, hero banners |

> `rem` is generally preferred for font sizes and spacing because it's consistent across the entire page.

---

### Q9: What is the difference between `position: relative`, `absolute`, `fixed`, and `sticky`?

**Ideal Answer:**

> `relative` — Element stays in normal flow but can be offset with `top/left/right/bottom`. Creates a positioning context for absolute children. `absolute` — Element is removed from normal flow and positioned relative to the nearest positioned ancestor. `fixed` — Element is removed from flow and positioned relative to the viewport; it stays in place when scrolling. `sticky` — Acts like `relative` until a scroll threshold is reached, then becomes `fixed`.

```css
.tooltip {
  position: absolute;  /* Positioned relative to its closest positioned ancestor */
  top: -30px;
  left: 0;
}

.navbar {
  position: sticky;  /* Sticks to top when you scroll past it */
  top: 0;
}
```

---

### Q10: What is the `z-index` and how does it work?

**Ideal Answer:**

> `z-index` controls the stacking order of positioned elements (elements with a `position` value other than `static`). Higher values appear on top of lower values. It only works on positioned elements. A common pitfall is stacking context — a parent with a `z-index` creates a new stacking context, so children can't appear above elements outside that context regardless of their z-index value.

---

### Q11: What are pseudo-classes and pseudo-elements? Give examples.

**Ideal Answer:**

> Pseudo-classes select elements based on their **state**: `:hover` (mouse over), `:focus` (keyboard focus), `:first-child`, `:nth-child(n)`, `:disabled`, `:checked`. They use a single colon. Pseudo-elements select a **part** of an element: `::before` (inserts content before), `::after` (inserts content after), `::first-line`, `::placeholder`. They use double colons.

```css
/* Pseudo-class: state-based */
button:hover { background-color: #333; }
input:focus { border-color: blue; }
tr:nth-child(even) { background-color: #f5f5f5; }

/* Pseudo-element: part of element */
.required-label::after {
  content: " *";
  color: red;
}
```

---

### Q12: What is the difference between `display: none` and `visibility: hidden`?

**Ideal Answer:**

> `display: none` completely removes the element from the document flow — it takes up no space and is invisible. `visibility: hidden` makes the element invisible but it still occupies space in the layout. Use `display: none` when you want the element gone entirely (like toggling a menu). Use `visibility: hidden` when you want to preserve the layout space (like hiding a placeholder temporarily).

---

### Q13: What is a CSS preprocessor? Name some examples.

**Ideal Answer:**

> CSS preprocessors extend CSS with features like variables, nesting, mixins, and functions that get compiled into standard CSS. Examples: **SASS/SCSS** (most popular), **LESS**, and **Stylus**. Modern CSS now has native variables (`--custom-property`) and nesting, reducing the need for preprocessors, but they're still widely used in production codebases.

---

### Q14: How do CSS transitions and animations differ?

**Ideal Answer:**

> Transitions animate property changes between two states — they need a trigger (like `:hover`). Animations can run automatically, loop, and have multiple keyframes for complex multi-step sequences.

```css
/* Transition: simple hover effect */
.button {
  background: blue;
  transition: background 0.3s ease, transform 0.2s ease;
}
.button:hover {
  background: darkblue;
  transform: scale(1.05);
}

/* Animation: complex, auto-playing */
@keyframes fadeIn {
  from { opacity: 0; transform: translateY(20px); }
```

### Q1: What is the difference between `var`, `let`, and `const`?

**Ideal Answer:**

> `var` is function-scoped, gets hoisted with value `undefined`, and can be redeclared. `let` is block-scoped, hoisted but not initialized (temporal dead zone), and can be reassigned but not redeclared. `const` is block-scoped, hoisted but not initialized, and cannot be reassigned after declaration. **Best practice:** Use `const` by default, `let` when you need to reassign, never use `var`.

```javascript
// var — function-scoped (leaks out of blocks)
if (true) { var x = 10; }
console.log(x); // 10 ← var leaks outside the if block

// let — block-scoped
if (true) { let y = 10; }
console.log(y); // ReferenceError ← let stays inside the block

// const — cannot reassign
const z = 10;
z = 20; // TypeError: Assignment to constant variable

// BUT const objects/arrays CAN be mutated
const arr = [1, 2, 3];
arr.push(4);  // ✅ This works — you're mutating, not reassigning
arr = [5, 6]; // ❌ TypeError — this is reassigning
```

### Q2: Explain hoisting in JavaScript.

**Ideal Answer:**

> Hoisting is JavaScript's behavior of moving declarations to the top of their scope during compilation. `var` declarations are hoisted and initialized to `undefined`. `let` and `const` are hoisted but NOT initialized — accessing them before declaration causes a ReferenceError (temporal dead zone). Function declarations are fully hoisted (both name and body). Function expressions are NOT fully hoisted.

```javascript
console.log(a); // undefined (var is hoisted, initialized to undefined)
var a = 5;

console.log(b); // ReferenceError (let is in temporal dead zone)
let b = 5;

greet(); // "Hello!" (function declarations are fully hoisted)
function greet() { console.log("Hello!"); }

sayHi(); // TypeError: sayHi is not a function
var sayHi = function() { console.log("Hi!"); };
// var sayHi is hoisted as undefined, but the function assignment is not
```

### Q3: What are closures? Give a practical example.

**Ideal Answer:**

> A closure is a function that retains access to variables from its outer (enclosing) function's scope, even after the outer function has finished executing. Closures are created every time a function is defined inside another function.

```javascript
// Practical example: Counter factory
function createCounter() {
  let count = 0;  // This variable is "closed over"
  
  return {
    increment: function() { count++; return count; },
    decrement: function() { count--; return count; },
    getCount: function() { return count; }
  };
}

const counter = createCounter();
counter.increment(); // 1
counter.increment(); // 2
counter.decrement(); // 1
counter.getCount();  // 1

// 'count' is private — you can't access it directly
console.log(counter.count); // undefined
```

> **Real-world use cases:** Data privacy/encapsulation, event handlers, setTimeout callbacks, module pattern.

**Common mistake:** Not understanding that closures capture variables by reference, not by value.

### Q4: What is the difference between `==` and `===`?

**Ideal Answer:**

> `==` (loose equality) performs type coercion before comparing — it converts values to the same type first. `===` (strict equality) compares both value AND type without coercion. **Always use `===`** to avoid unexpected behavior.

```javascript

0 === ""      // false (number vs string)
null == undefined  // true  (special case)
null === undefined // false (different types)
false == 0    // true  (both coerced to 0)
false === 0   // false (boolean vs number)
```

### Q5: Explain the event loop in JavaScript.

**Ideal Answer:**

> JavaScript is single-threaded — it can only execute one task at a time. The event loop enables asynchronous behavior by managing a call stack, a task queue (for callbacks like setTimeout), and a microtask queue (for Promises). When the call stack is empty, the event loop first processes all microtasks, then takes the next task from the task queue and pushes it onto the call stack.

```javascript
console.log("1");                      // Sync — runs immediately

setTimeout(() => console.log("2"), 0); // Goes to task queue

Promise.resolve().then(() => console.log("3")); // Goes to microtask queue

console.log("4");                      // Sync — runs immediately

// Output: 1, 4, 3, 2
// Sync first, then microtasks (Promise), then task queue (setTimeout)
```

**Follow-up:** "Why does setTimeout with 0ms not execute immediately?" → Because it's placed in the task queue and must wait for the call stack to be empty AND all microtasks to complete.

### Q6: What are Promises? How do they work?

**Ideal Answer:**

> A Promise represents a value that may be available now, later, or never. It has three states: **pending** (initial), **fulfilled** (resolved successfully), **rejected** (failed). You handle results with `.then()` for success and `.catch()` for errors.

```javascript
// Creating a Promise
function fetchUser(id) {
  return new Promise((resolve, reject) => {
    setTimeout(() => {
      if (id > 0) {
        resolve({ id, name: "Juan" });
      } else {
        reject(new Error("Invalid ID"));
      }
    }, 1000);
  });
}

// Using a Promise
fetchUser(1)
  .then(user => console.log(user.name))  // "Juan"
  .catch(error => console.error(error))
  .finally(() => console.log("Done"));   // Always runs
```

### Q7: Explain `async/await` and how it relates to Promises.

**Ideal Answer:**

> `async/await` is syntactic sugar over Promises that makes asynchronous code look synchronous and easier to read. An `async` function always returns a Promise. `await` pauses execution until the Promise resolves. Error handling uses `try/catch` instead of `.catch()`.

```javascript
// With Promises (callback-style)

  fetch('/api/users/1')
    .then(res => res.json())
    .then(user => {
      return fetch(`/api/posts?userId=${user.id}`);
    })
    .then(res => res.json())
    .then(posts => console.log(posts))
    .catch(err => console.error(err));
}

// With async/await (cleaner)
async function getUser() {
  try {
    const res = await fetch('/api/users/1');
    const user = await res.json();
    const postsRes = await fetch(`/api/posts?userId=${user.id}`);
    const posts = await postsRes.json();
    console.log(posts);
  } catch (err) {
    console.error(err);
  }
}
```

**Follow-up:** "Can you use `await` outside an `async` function?" → Yes, with top-level await in ES modules, but not in regular scripts.

### Q8: What are arrow functions and how do they differ from regular functions?

**Ideal Answer:**

(2) They can't be used as constructors (`new`). (3) They don't have `arguments` object. (4) They have implicit return for single expressions.

```javascript

// Arrow function (concise)

// this binding difference

  name: "Juan",
  // Regular: 'this' refers to obj
  greet: function() { console.log(this.name); },
  // Arrow: 'this' refers to outer scope (probably window/undefined)
  greetArrow: () => { console.log(this.name); }
};
obj.greet();      // "Juan"
obj.greetArrow(); // undefined
```

### Q20: What is the nullish coalescing operator (`??`)?

### Q21: What is `Promise.all()` vs `Promise.allSettled()`?

**Ideal Answer:**

> `Promise.all()` runs multiple promises in parallel and resolves when ALL succeed — but rejects immediately if ANY one fails. `Promise.allSettled()` waits for ALL promises to complete regardless of success/failure and returns the result of each.

```javascript
// Promise.all — fails fast
const [users, posts] = await Promise.all([
  fetch('/api/users').then(r => r.json()),
  fetch('/api/posts').then(r => r.json())
]);
// If either fails, both fail

// Promise.allSettled — always completes
const results = await Promise.allSettled([
  fetch('/api/users').then(r => r.json()),
  fetch('/api/posts').then(r => r.json())
]);
results.forEach(result => {
  if (result.status === 'fulfilled') console.log(result.value);
  else console.log('Failed:', result.reason);
});
```

### Q22: What is debouncing and throttling?

**Ideal Answer:**

```javascript
  };
}
```

> JSX is a syntax extension for JavaScript that lets you write HTML-like code inside JavaScript. It's not valid JavaScript — Babel transpiles it into `React.createElement()` calls. Key rules: (1) Must return a single parent element (use `<>` fragments). (2) Use `className` instead of `class`. (3) Use `htmlFor` instead of `for`. (4) All tags must be closed. (5) JavaScript expressions use curly braces `{}`.

```jsx
// JSX
const element = <h1 className="title">Hello, {name}!</h1>;

// Compiles to
const element = React.createElement('h1', { className: 'title' }, `Hello, ${name}!`);
```

### Q3: What is the difference between functional and class components?

**Ideal Answer:**

> Modern React uses **functional components** almost exclusively. They're simpler functions that accept props and return JSX. Class components use ES6 classes with a `render()` method, lifecycle methods, and `this.state`. Functional components use hooks (`useState`, `useEffect`) instead of lifecycle methods. Unless maintaining legacy code, you should always use functional components.

```jsx
// Functional component (modern)
function Greeting({ name }) {
  return <h1>Hello, {name}!</h1>;
}

// Class component (legacy)
class Greeting extends React.Component {
  render() {
    return <h1>Hello, {this.props.name}!</h1>;
  }
}
```

### Q4: What are props and state?

**Ideal Answer:**

> **Props** are inputs passed from parent to child — they are read-only and the child cannot modify them. **State** is internal data managed within the component — it can be updated and causes re-renders when changed. Think of props as function arguments and state as local variables.

```jsx
function UserCard({ name, role }) {     // name and role are props
  const [isOnline, setIsOnline] = useState(false); // isOnline is state
  
  return (
    <div>
      <h2>{name}</h2>          {/* From props — read-only */}
      <p>{role}</p>
      <p>{isOnline ? "🟢 Online" : "⚫ Offline"}</p>
      <button onClick={() => setIsOnline(!isOnline)}>Toggle</button>
    </div>
  );
}

// Usage:
<UserCard name="Juan" role="Intern" />
```

### Q5: Explain `useState` hook.

**Ideal Answer:**

> `useState` adds state to functional components. It returns an array with the current state value and a setter function. When the setter is called, React re-renders the component with the new value.

```jsx
function Counter() {
  const [count, setCount] = useState(0);  // Initial value = 0

  return (
    <div>
      <p>Count: {count}</p>
      <button onClick={() => setCount(count + 1)}>+</button>
      <button onClick={() => setCount(prev => prev - 1)}>-</button>
      {/* Use functional update (prev => ...) when new state depends on old state */}
    </div>
  );
}
```

**Common mistake:** Directly mutating state — `count++` doesn't work. You must use the setter function.

### Q6: Explain `useEffect` hook.

**Ideal Answer:**

> `useEffect` handles side effects in functional components — data fetching, subscriptions, DOM manipulation, timers. It runs after every render by default. The dependency array controls when it runs.

```jsx
function UserProfile({ userId }) {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    // Runs when userId changes
    setLoading(true);
    fetch(`/api/users/${userId}`)
      .then(res => res.json())
      .then(data => {
        setUser(data);
        setLoading(false);
      });
      
    // Cleanup function (runs on unmount or before re-run)
    return () => {
      console.log("Cleanup: cancelled fetch for", userId);
    };
  }, [userId]); // Dependency array

  if (loading) return <p>Loading...</p>;
  return <h1>{user.name}</h1>;
}
```

> **Dependency array patterns:**
> - `useEffect(() => {}, [])` — Runs ONCE on mount (like componentDidMount)
> - `useEffect(() => {}, [dep])` — Runs when `dep` changes
> - `useEffect(() => {})` — Runs on EVERY render (usually a mistake)

### Q9: What is the `key` prop in lists and why is it important?

**Ideal Answer:**

> The `key` prop helps React identify which items in a list have changed, been added, or removed. It should be a unique, stable identifier (like a database ID), NOT the array index. Without proper keys, React can't efficiently update the DOM and may cause bugs with component state.

```jsx
// ✅ Good — using unique ID
{users.map(user => (
  <UserCard key={user.id} name={user.name} />
))}

// ❌ Bad — using index (causes bugs when reordering)
{users.map((user, index) => (
  <UserCard key={index} name={user.name} />
))}
```

### Q16: What is the Virtual DOM?

**Ideal Answer:**

### Q17: What is `useMemo` and `useCallback`?

**Ideal Answer:**

> `useMemo` memoizes a computed value — recalculates only when dependencies change. Used for expensive calculations. `useCallback` memoizes a function reference — creates a new function only when dependencies change. Used when passing callbacks to child components that use `React.memo`.

```jsx
// useMemo — avoid re-computing on every render
const filteredUsers = useMemo(() => {
  return users.filter(u => u.name.includes(searchTerm));
}, [users, searchTerm]);

// useCallback — avoid re-creating function on every render
const handleDelete = useCallback((id) => {
  setUsers(prev => prev.filter(u => u.id !== id));
}, []);
```

### Q18: Coding — Build a Todo App Component.

**Ideal Answer:**

```jsx
function TodoApp() {
  const [todos, setTodos] = useState([]);
  const [input, setInput] = useState('');

  const addTodo = (e) => {
    e.preventDefault();
    if (!input.trim()) return;
    setTodos(prev => [...prev, { id: Date.now(), text: input, done: false }]);
    setInput('');

  const toggleTodo = (id) => {

    setTodos(prev => prev.filter(t => t.id !== id));
  };

  return (
    <div>
      <h1>Todo App</h1>
      <form onSubmit={addTodo}>
        <input value={input} onChange={e => setInput(e.target.value)} placeholder="Add task..." />
        <button type="submit">Add</button>
      </form>
      <ul>
        {todos.map(todo => (
          <li key={todo.id} style={{ textDecoration: todo.done ? 'line-through' : 'none' }}>
            <span onClick={() => toggleTodo(todo.id)}>{todo.text}</span>
            <button onClick={() => deleteTodo(todo.id)}>Delete</button>
          </li>
        ))}
      </ul>
      <p>{todos.filter(t => !t.done).length} tasks remaining</p>
    </div>
  );
}
```

### Q19: Debugging — Why isn't this component updating?

```jsx
function UserList() {
  const [users, setUsers] = useState([{ id: 1, name: "Juan" }]);
  
  const addUser = () => {
    users.push({ id: 2, name: "Maria" });
    setUsers(users);
  };

  return <button onClick={addUser}>Add User</button>;
}
```

**Ideal Answer:**

> The component doesn't update because `users.push()` mutates the existing array, and `setUsers(users)` passes the same reference. React uses reference equality to detect changes — since the array reference hasn't changed, React thinks nothing updated. Fix: Create a new array.

```jsx
const addUser = () => {
  setUsers(prev => [...prev, { id: 2, name: "Maria" }]);
};
```

### Q21: What is React.memo?

**Ideal Answer:**

> `React.memo` is a higher-order component that prevents re-rendering when props haven't changed. By default, when a parent re-renders, ALL children re-render. Wrapping a child in `React.memo` makes it skip re-rendering if its props are the same.

```jsx
const UserCard = React.memo(function UserCard({ name, role }) {

});
```

### Q1: What is PHP and where does it run?

### Q2: What is the difference between `GET` and `POST` in PHP?

**Ideal Answer:**

> `$_GET` retrieves data sent via URL query parameters — visible in the address bar, limited in size, bookmarkable, used for fetching/filtering data. `$_POST` retrieves data sent in the request body — hidden from URL, no size limit, used for submitting forms that create or modify data (login, registration). Never send passwords or sensitive data via GET.

```php
// GET: /search.php?query=php&page=2
$query = $_GET['query'];  // "php"
$page = $_GET['page'];    // "2"

// POST: from a form with method="POST"
$email = $_POST['email'];
$password = $_POST['password'];
```

### Q4: What are prepared statements and why are they important?

**Ideal Answer:**

> Prepared statements separate SQL logic from data, preventing SQL injection attacks. The database compiles the query first, then fills in the parameters separately — so user input can never be interpreted as SQL code.

```php
// ❌ VULNERABLE to SQL injection
$sql = "SELECT * FROM users WHERE email = '$email'";
// If $email = "' OR '1'='1", it returns ALL users

// ✅ SAFE with prepared statements
$stmt = $pdo->prepare("SELECT * FROM users WHERE email = :email");
$stmt->execute(['email' => $email]);
$user = $stmt->fetch();
```

**This is one of the most important PHP security questions. Always use prepared statements.**

### Q6: What are PHP sessions? How do they work?

**Ideal Answer:**

> Sessions store user data on the server between page requests. When a session starts, PHP generates a unique session ID stored in a cookie (PHPSESSID). Each request sends this cookie, and the server retrieves the associated data.

```php
// Starting a session (must be at the top of every page that uses sessions)
session_start();

// Setting session data (login)
$_SESSION['user_id'] = $user['id'];
$_SESSION['user_name'] = $user['name'];
$_SESSION['logged_in'] = true;

// Checking if logged in (on protected pages)
if (!isset($_SESSION['logged_in']) || $_SESSION['logged_in'] !== true) {
    header('Location: login.php');
    exit();
}

// Destroying session (logout)
session_start();
$_SESSION = [];
session_destroy();
header('Location: login.php');
```

### Q9: What is CSRF and how do you prevent it?

**Ideal Answer:**

> CSRF (Cross-Site Request Forgery) is an attack where a malicious website tricks a logged-in user into making unwanted requests to your site. Prevention: Generate a random token, store it in the session, include it as a hidden field in forms, and verify it on submission.

```php
// Generate token
session_start();
$_SESSION['csrf_token'] = bin2hex(random_bytes(32));

// In the form
echo '<input type="hidden" name="csrf_token" value="' . $_SESSION['csrf_token'] . '">';

// On submission
if ($_POST['csrf_token'] !== $_SESSION['csrf_token']) {
    die("CSRF token mismatch");
}
```

### Q11: What is password hashing and why should you never store plain text passwords?

**Ideal Answer:**

> Plain text passwords mean that if the database is breached, all passwords are exposed. PHP provides `password_hash()` which uses bcrypt to create a one-way hash. Even if the hash is stolen, the original password can't be reversed.

```php
// Storing password (during registration)
$hashedPassword = password_hash($password, PASSWORD_BCRYPT);
// Store $hashedPassword in database

// Verifying password (during login)
$stmt = $pdo->prepare("SELECT * FROM users WHERE email = :email");
$stmt->execute(['email' => $email]);
$user = $stmt->fetch();

if ($user && password_verify($password, $user['password'])) {
    // Login successful
    $_SESSION['user_id'] = $user['id'];
} else {
    // Invalid credentials
    echo "Invalid email or password";
}
```

### Q12: What is XSS and how do you prevent it in PHP?

**Ideal Answer:**

> XSS (Cross-Site Scripting) is when an attacker injects malicious JavaScript into your page through user input. Prevention: Always escape output using `htmlspecialchars()`.

```php
// ❌ VULNERABLE
echo "<p>Welcome, " . $_GET['name'] . "</p>";

// ✅ SAFE
echo "<p>Welcome, " . htmlspecialchars($_GET['name'], ENT_QUOTES, 'UTF-8') . "</p>";
// Converts < > to &lt; &gt; so it displays as text, not code
```

```php
// Looping

}
```

### Q19: What is the difference between `isset()`, `empty()`, and `is_null()`?

|----------|-------------------|

| `empty($x)` | Variable is `""`, `0`, `"0"`, `null`, `false`, `[]`, or not set |

# SECTION 7: POSTGRESQL & SQL

> SQL questions are a staple of Philippine internship interviews. Expect query writing, joins, normalization, and aggregate functions. Many companies include SQL questions in written exams.

---

### Q1: What is PostgreSQL and how is it different from MySQL?

**Ideal Answer:**

> PostgreSQL is an advanced, open-source relational database known for its standards compliance, extensibility, and support for complex queries. Key differences from MySQL: PostgreSQL supports JSON/JSONB data types natively, has better support for CTEs (Common Table Expressions), supports table inheritance, has stricter SQL standard compliance, and handles concurrent writes better with MVCC. MySQL is simpler to set up and traditionally faster for read-heavy workloads.

---

### Q2: What is a primary key?

**Ideal Answer:**

> A primary key uniquely identifies each record in a table. It must be unique and cannot be null. Each table should have exactly one primary key. It can be a single column or a combination of columns (composite key). PostgreSQL automatically creates an index on the primary key.

```sql
CREATE TABLE users (
  id SERIAL PRIMARY KEY,  -- Auto-incrementing integer
  email VARCHAR(255) UNIQUE NOT NULL,
  name VARCHAR(100) NOT NULL
);
```

---

### Q3: What is a foreign key?

**Ideal Answer:**

> A foreign key is a column that references the primary key of another table, creating a relationship between them. It enforces referential integrity — you can't insert a foreign key value that doesn't exist in the referenced table, and you can't delete a referenced row without handling dependent records.

```sql
CREATE TABLE posts (
  id SERIAL PRIMARY KEY,
  title VARCHAR(255) NOT NULL,
  user_id INTEGER NOT NULL,
  FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);
-- ON DELETE CASCADE: when a user is deleted, their posts are automatically deleted
-- ON DELETE SET NULL: sets the foreign key to NULL
-- ON DELETE RESTRICT: prevents deletion (default)
```

---

### Q4: Explain the different types of JOINs.

**Ideal Answer:**

> **INNER JOIN** — Returns rows that have matching values in both tables. **LEFT JOIN** — Returns all rows from the left table, with NULLs for non-matching right table rows. **RIGHT JOIN** — Returns all rows from the right table, with NULLs for non-matching left table rows. **FULL OUTER JOIN** — Returns all rows from both tables, with NULLs where there's no match.

```sql
-- Sample data:
-- users: (1, 'Juan'), (2, 'Maria'), (3, 'Pedro')
-- posts: (1, 'Post A', user_id=1), (2, 'Post B', user_id=1), (3, 'Post C', user_id=4)

-- INNER JOIN: Only users WITH posts (and valid user_ids)
SELECT u.name, p.title
FROM users u
INNER JOIN posts p ON u.id = p.user_id;
-- Result: Juan - Post A, Juan - Post B

-- LEFT JOIN: ALL users, even those without posts
SELECT u.name, p.title
FROM users u
LEFT JOIN posts p ON u.id = p.user_id;
-- Result: Juan - Post A, Juan - Post B, Maria - NULL, Pedro - NULL
```

---

### Q5: What is normalization? Explain 1NF, 2NF, 3NF.

**Ideal Answer:**

> Normalization is organizing database tables to reduce redundancy and improve data integrity.
> - **1NF** — Each column contains atomic (indivisible) values; no repeating groups. E.g., don't store "PHP, JS, React" in one column.
> - **2NF** — Must be in 1NF + every non-key column depends on the entire primary key (not just part of a composite key).
> - **3NF** — Must be in 2NF + no transitive dependencies (non-key columns don't depend on other non-key columns). E.g., don't store both `department_id` and `department_name` in the employees table — store `department_name` only in the departments table.

---

### Q6: What is an index? When should you use one?

**Ideal Answer:**

> An index is a data structure that speeds up data retrieval by creating a sorted reference to rows. Like a book index — instead of reading every page, you look up the topic and jump to the right page. Use indexes on columns frequently used in WHERE, JOIN, and ORDER BY clauses. Don't over-index — they slow down INSERT, UPDATE, and DELETE operations.

```sql
-- Create index on email (frequently searched)
CREATE INDEX idx_users_email ON users(email);

-- Composite index for queries filtering by both columns
CREATE INDEX idx_posts_user_status ON posts(user_id, status);

-- Unique index (also enforces uniqueness)
CREATE UNIQUE INDEX idx_users_email_unique ON users(email);
```

### Q7: What are aggregate functions? List and explain them.

**Ideal Answer:**

```sql
-- COUNT: Number of rows
SELECT COUNT(*) FROM users;                    -- Total users
SELECT COUNT(DISTINCT department_id) FROM users; -- Unique departments

-- SUM: Total of numeric column
SELECT SUM(salary) FROM employees;

-- AVG: Average value
SELECT AVG(salary) FROM employees;

-- MAX / MIN: Highest / lowest value
SELECT MAX(salary), MIN(salary) FROM employees;

-- Combined with GROUP BY
SELECT department, COUNT(*) as employee_count, AVG(salary) as avg_salary
FROM employees
GROUP BY department
HAVING AVG(salary) > 50000;  -- HAVING filters grouped results (WHERE filters before grouping)
```

---

### Q8: What is the difference between WHERE and HAVING?

**Ideal Answer:**

> `WHERE` filters rows **before** grouping (works on individual rows). `HAVING` filters groups **after** GROUP BY (works on aggregated results). You can't use aggregate functions in WHERE.

```sql
-- WHERE: filter before grouping
SELECT department, AVG(salary)
FROM employees
WHERE status = 'active'        -- Filters individual rows first
GROUP BY department;

-- HAVING: filter after grouping
SELECT department, AVG(salary) as avg_sal
FROM employees
GROUP BY department
HAVING AVG(salary) > 50000;    -- Filters the grouped results
```

---

### Q9: What is a subquery?

**Ideal Answer:**

```sql
-- Scalar subquery in WHERE
SELECT name, salary
FROM employees
WHERE salary > (SELECT AVG(salary) FROM employees);

-- Subquery in FROM (derived table)
SELECT dept_name, avg_salary
FROM (
  SELECT department AS dept_name, AVG(salary) AS avg_salary
  FROM employees
  GROUP BY department
) AS dept_stats
WHERE avg_salary > 50000;

-- IN subquery
SELECT name FROM users
WHERE id IN (SELECT DISTINCT user_id FROM orders WHERE total > 1000);
```

### Q10: What are CTEs (Common Table Expressions)?

**Ideal Answer:**

```sql
-- CTE makes complex queries readable
WITH active_users AS (
  SELECT id, name, email
  FROM users
  WHERE status = 'active'
),
user_orders AS (
  SELECT user_id, COUNT(*) as order_count, SUM(total) as total_spent
  FROM orders
  GROUP BY user_id
)
SELECT au.name, au.email, uo.order_count, uo.total_spent
FROM active_users au
LEFT JOIN user_orders uo ON au.id = uo.user_id
ORDER BY uo.total_spent DESC NULLS LAST;
```

### Q11: What is a transaction?

**Ideal Answer:**

> A transaction groups multiple SQL operations into a single atomic unit — either ALL succeed or ALL fail. This ensures data consistency. ACID properties: Atomicity (all or nothing), Consistency (valid state), Isolation (transactions don't interfere), Durability (committed data persists).

```sql
BEGIN;

UPDATE accounts SET balance = balance - 1000 WHERE id = 1;
UPDATE accounts SET balance = balance + 1000 WHERE id = 2;

-- If both succeed
COMMIT;

-- If anything fails
ROLLBACK;
```

# SECTION 8: PRISMA ORM

> Prisma is a modern ORM for Node.js. If the company uses it, expect questions about schema definition, migrations, and CRUD operations with Prisma Client.

### Q1: What is Prisma and why is it used?

**Ideal Answer:**

> Prisma is a next-generation ORM (Object-Relational Mapping) for Node.js and TypeScript. It provides three main tools: (1) **Prisma Schema** — a declarative data model definition. (2) **Prisma Migrate** — database migration system. (3) **Prisma Client** — auto-generated, type-safe database client. Companies use it because it provides type safety, auto-completion, and prevents writing raw SQL for most operations.

### Q2: What does a Prisma schema look like?

**Ideal Answer:**

```prisma
// prisma/schema.prisma

generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

model User {
  id        Int      @id @default(autoincrement())
  email     String   @unique
  name      String
  role      Role     @default(USER)
  posts     Post[]          // One-to-many: User has many Posts
  profile   Profile?        // One-to-one: User has one optional Profile
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  @@map("users")            // Maps to "users" table in database
}

model Post {
  id        Int      @id @default(autoincrement())
  title     String
  content   String?
  published Boolean  @default(false)
  author    User     @relation(fields: [authorId], references: [id])
  authorId  Int
  tags      Tag[]           // Many-to-many
  createdAt DateTime @default(now())

  @@index([authorId])       // Index for faster queries
  @@map("posts")
}

model Profile {
  id     Int    @id @default(autoincrement())
  bio    String?
  avatar String?
  user   User   @relation(fields: [userId], references: [id], onDelete: Cascade)
  userId Int    @unique
}

model Tag {
  id    Int    @id @default(autoincrement())
  name  String @unique
  posts Post[]
}

enum Role {
  USER
  ADMIN
  MODERATOR
}
```

### Q5: What are the different types of relations in Prisma?

**Ideal Answer:**

> **One-to-One** — User has one Profile. One side has `@unique` on the foreign key. **One-to-Many** — User has many Posts. The "many" side holds the foreign key. **Many-to-Many** — Posts have many Tags and Tags have many Posts. Prisma creates a junction table automatically (implicit) or you define it (explicit).

```prisma
// One-to-One
model User {
  profile Profile?  // Optional one-to-one
}
model Profile {
  user   User @relation(fields: [userId], references: [id])
  userId Int  @unique
}

// One-to-Many
model User {
  posts Post[]      // User has many posts
}
model Post {
  author   User @relation(fields: [authorId], references: [id])
  authorId Int
}

// Many-to-Many (implicit — Prisma creates junction table)
model Post {
  tags Tag[]
}
model Tag {
  posts Post[]
}
```

### Q6: What is `findFirst` vs `findUnique`?

**Ideal Answer:**

> `findUnique` queries by unique fields only (id, email with @unique) and uses the unique index for fast lookups. `findFirst` can query by any field and returns the first matching record — it's like `findMany` with `take: 1`. Use `findUnique` when you have a unique identifier; `findFirst` when searching by non-unique fields.

### Q9: How do you perform transactions in Prisma?

```typescript
// Interactive transaction
const result = await prisma.$transaction(async (tx) => {

    where: { id: 1 },
```

### Q10: What is `select` vs `include` in Prisma?

**Ideal Answer:**

> `include` fetches all fields of the model PLUS the specified relations. `select` lets you pick specific fields — only selected fields are returned. You can't use both `select` and `include` at the same level.

```typescript

const user = await prisma.user.findUnique({
  where: { id: 1 },
  select: {
    name: true,
    email: true,
    posts: { select: { title: true } }
  }
});
// Returns: { name: "Juan", email: "...", posts: [{ title: "..." }] }
```

### Q4: What are Controllers?

**Ideal Answer:**

> Controllers handle incoming HTTP requests and return responses. They use decorators to define routes and HTTP methods. Controllers should be thin — they delegate business logic to services.

```typescript
@Controller('users')  // Base route: /users
export class UsersController {
  constructor(private readonly usersService: UsersService) {}

  @Get()                    // GET /users
  findAll() {
    return this.usersService.findAll();
  }

  @Get(':id')               // GET /users/:id
  findOne(@Param('id', ParseIntPipe) id: number) {
    return this.usersService.findOne(id);
  }

  @Post()                   // POST /users
  create(@Body() createUserDto: CreateUserDto) {
    return this.usersService.create(createUserDto);
  }

  @Patch(':id')             // PATCH /users/:id
  update(@Param('id', ParseIntPipe) id: number, @Body() updateUserDto: UpdateUserDto) {
    return this.usersService.update(id, updateUserDto);
  }

  @Delete(':id')            // DELETE /users/:id
  remove(@Param('id', ParseIntPipe) id: number) {
    return this.usersService.remove(id);
  }
}
```

### Q5: What are Services?

**Ideal Answer:**

> Services contain business logic and are decorated with `@Injectable()`. They're injected into controllers via dependency injection. This separation means controllers handle HTTP concerns while services handle data operations and business rules.

```typescript
@Injectable()
export class UsersService {
  constructor(private readonly prisma: PrismaService) {}

  async findAll() {
    return this.prisma.user.findMany({
      include: { profile: true }
    });
  }

  async findOne(id: number) {
    const user = await this.prisma.user.findUnique({
      where: { id },
      include: { posts: true }
    });
    if (!user) throw new NotFoundException(`User #${id} not found`);
    return user;
  }

  async create(data: CreateUserDto) {
    try {
      return await this.prisma.user.create({ data });
    } catch (error) {
      if (error.code === 'P2002') {
        throw new ConflictException('Email already exists');
      }
      throw error;
    }
  }

  async update(id: number, data: UpdateUserDto) {
    await this.findOne(id);  // Throws if not found
    return this.prisma.user.update({ where: { id }, data });
  }

  async remove(id: number) {
    await this.findOne(id);
    return this.prisma.user.delete({ where: { id } });
  }
}
```

### Q9: What are Guards?

**Ideal Answer:**

> Guards determine whether a request should be handled — mainly used for authentication and authorization. They run before the route handler and return `true` (allow) or `false` (deny).

```typescript
@Injectable()
export class AuthGuard implements CanActivate {
  constructor(private readonly jwtService: JwtService) {}

  canActivate(context: ExecutionContext): boolean {
    const request = context.switchToHttp().getRequest();
    const token = request.headers.authorization?.split(' ')[1];

    if (!token) throw new UnauthorizedException('No token provided');

    try {
      const payload = this.jwtService.verify(token);
      request.user = payload;
      return true;
    } catch {
      throw new UnauthorizedException('Invalid token');
    }
  }
}

// Usage
@UseGuards(AuthGuard)
@Get('profile')
getProfile(@Request() req) {
  return req.user;
}
```

### Q13: How do you handle exceptions in NestJS?

**Ideal Answer:**

```typescript
// Custom exception filter
@Catch()
export class AllExceptionsFilter implements ExceptionFilter {
  catch(exception: unknown, host: ArgumentsHost) {
    const ctx = host.switchToHttp();
    const response = ctx.getResponse();
    const status = exception instanceof HttpException 
      ? exception.getStatus() 
      : 500;

    response.status(status).json({
      statusCode: status,
      message: exception instanceof HttpException 
        ? exception.message 
        : 'Internal server error',
      timestamp: new Date().toISOString()
    });
  }
}
```

# SECTION 10: OOP

> OOP is fundamental and heavily tested. Interviewers want to see that you understand the four pillars and can apply them, not just recite definitions.

---

### Q1: What are the four pillars of OOP?

**Ideal Answer:**

> **Encapsulation** — Bundling data and methods together, hiding internal details. Like a TV remote — you press buttons (public interface) without knowing the electronics inside (private implementation).
>
> **Inheritance** — A class (child) can inherit properties and methods from another class (parent). Like how a `Dog` inherits common traits from `Animal`.
>
> **Polymorphism** — Objects of different classes can be treated as the same type and respond differently to the same method. Like how `dog.speak()` returns "Woof" and `cat.speak()` returns "Meow" — same method name, different behavior.
>
> **Abstraction** — Hiding complex implementation details and showing only necessary features. Like driving a car — you use the steering wheel and pedals without knowing how the engine works internally.

---

### Q2: What is Encapsulation? Give a code example.

**Ideal Answer:**

```javascript
class BankAccount {
  #balance;  // Private field (encapsulated)
  #owner;

  constructor(owner, initialBalance) {
    this.#owner = owner;
    this.#balance = initialBalance;
  }

  // Public interface — controlled access
  deposit(amount) {
    if (amount <= 0) throw new Error("Amount must be positive");
    this.#balance += amount;
    return this.getBalance();
  }

  withdraw(amount) {
    if (amount <= 0) throw new Error("Amount must be positive");
    if (amount > this.#balance) throw new Error("Insufficient funds");
    this.#balance -= amount;
    return this.getBalance();
  }

  getBalance() {
    return this.#balance;
  }
}

const account = new BankAccount("Juan", 1000);
account.deposit(500);     // ✅ Uses public method
account.withdraw(200);    // ✅ Uses public method
account.#balance = 999999; // ❌ SyntaxError — can't access private field
```

> **Real-world analogy:** ATM machine. You can check balance, deposit, withdraw (public methods), but you can't directly access the vault (private data).

---

### Q3: What is Inheritance? When should you use it?

**Ideal Answer:**

```javascript
class Animal {
  constructor(name, sound) {
    this.name = name;
    this.sound = sound;
  }

  speak() {
    return `${this.name} says ${this.sound}!`;
  }

  eat(food) {
    return `${this.name} eats ${food}`;
  }
}

class Dog extends Animal {
  constructor(name, breed) {
    super(name, "Woof");  // Call parent constructor
    this.breed = breed;
  }

  fetch(item) {
    return `${this.name} fetches the ${item}!`;
  }
}

class Cat extends Animal {
  constructor(name) {
    super(name, "Meow");
  }

  purr() {
    return `${this.name} purrs...`;
  }
}

const dog = new Dog("Bantay", "Aspin");
dog.speak();  // "Bantay says Woof!" (inherited)
dog.eat("rice"); // "Bantay eats rice" (inherited)
dog.fetch("ball"); // "Bantay fetches the ball!" (own method)
```

> **Use inheritance when:** There is a clear "is-a" relationship (Dog IS AN Animal).

### Q4: What is Polymorphism? Give a real-world example.

**Ideal Answer:**

```javascript
class Shape {
  area() {
    throw new Error("area() must be implemented by subclass");
  }
}

class Circle extends Shape {
  constructor(radius) {
    super();
    this.radius = radius;
  }
  area() { return Math.PI * this.radius ** 2; }
}

class Rectangle extends Shape {
  constructor(width, height) {
    super();
    this.width = width;
    this.height = height;
  }
  area() { return this.width * this.height; }
}

class Triangle extends Shape {
  constructor(base, height) {
    super();
    this.base = base;
    this.height = height;
  }
  area() { return 0.5 * this.base * this.height; }
}

// Polymorphism: same method call, different behavior
const shapes = [new Circle(5), new Rectangle(4, 6), new Triangle(3, 8)];
shapes.forEach(shape => console.log(shape.area()));
// 78.54, 24, 12 — each shape calculates area differently
```

> **Real-world analogy:** Payment processing. `payment.process()` works the same way whether it's a CreditCard, GCash, or BankTransfer — but each processes differently internally.

### Q5: What is Abstraction?

**Ideal Answer:**

> Abstraction means hiding complex details and exposing only what's necessary. In languages like Java/TypeScript, you use abstract classes and interfaces. In JavaScript, you simulate it with base classes that throw errors for unimplemented methods.

```typescript
// TypeScript example (more likely in NestJS interviews)
abstract class PaymentGateway {
  abstract processPayment(amount: number): Promise<boolean>;
  abstract refund(transactionId: string): Promise<boolean>;

  // Concrete method shared by all implementations
  formatAmount(amount: number): string {
    return `₱${amount.toFixed(2)}`;
  }
}

class GCashPayment extends PaymentGateway {
  async processPayment(amount: number): Promise<boolean> {
    // GCash-specific implementation
    console.log(`Processing GCash payment: ${this.formatAmount(amount)}`);
    return true;
  }

  async refund(transactionId: string): Promise<boolean> {
    console.log(`Refunding GCash transaction: ${transactionId}`);
    return true;
  }
}
```

### Q6: What is the difference between a class and an object?

**Ideal Answer:**

> A class is a **blueprint/template** — it defines the structure and behavior. An object is an **instance** of that class — a specific realization with actual data. Like a class `Student` is the template, and `new Student("Juan", "BSIT")` is a specific student object.

### Q7: What is the difference between Composition and Inheritance? When should you use each?

**Ideal Answer:**

> **Inheritance** = "is-a" relationship (Dog IS AN Animal). **Composition** = "has-a" relationship (Car HAS AN Engine). Composition is often preferred because it's more flexible — you can swap components at runtime.

```javascript
// Composition example (preferred)
class Engine {
  start() { return "Engine started"; }
}

class GPS {
  navigate(destination) { return `Navigating to ${destination}`; }
}

class Car {
  constructor() {
    this.engine = new Engine();  // Car HAS an Engine
    this.gps = new GPS();        // Car HAS a GPS
  }

  drive(destination) {
    this.engine.start();
    this.gps.navigate(destination);
  }
}
```

> **Rule of thumb:** Favor composition over inheritance. Only use inheritance when there's a clear, stable "is-a" hierarchy.

### Q11: What is the `static` keyword?

**Ideal Answer:**

# SECTION 11: MVC & MVVM

> Philippine companies commonly ask about architecture patterns, especially the differences between MVC and MVVM and when to use each.

---

### Q1: What is MVC?

**Ideal Answer:**

> MVC (Model-View-Controller) separates an application into three components:
> - **Model** — Data and business logic (database queries, validation)
> - **View** — UI presentation (what the user sees)
> - **Controller** — Handles user input, updates the Model, and selects the View
>
> The flow: User interacts with View → Controller processes the request → Controller updates Model → Model notifies View → View updates.

> **Real-world example:** In PHP — Model: database queries. View: HTML templates. Controller: PHP files that handle form submissions and route to the correct view.

---

### Q2: What is MVVM?

**Ideal Answer:**

> MVVM (Model-View-ViewModel) separates an application into:
> - **Model** — Data and business logic (same as MVC)
> - **View** — UI presentation (same as MVC)
> - **ViewModel** — Acts as a bridge between Model and View with two-way data binding. When the ViewModel changes, the View automatically updates, and vice versa.
>
> **Real-world example:** React with state management. The View is JSX, the ViewModel is your component with useState/useContext (manages data the View binds to), and the Model is the API/database layer.

---

### Q3: What is the difference between MVC and MVVM?

**Ideal Answer:**

| Aspect | MVC | MVVM |
|--------|-----|------|
| Binding | Manual (Controller updates View) | Two-way data binding (automatic) |
| Controller/ViewModel | Controller processes input | ViewModel exposes data to View |
| Communication | Controller → View (one-directional) | ViewModel ↔ View (bidirectional) |
| Best for | Server-rendered apps (PHP, NestJS) | Client-side apps (React, Angular) |
| Testability | Controller is testable | ViewModel is highly testable |
| Example | Laravel, NestJS, Express | React, Angular, Vue |

---

### Q4: How does MVC apply to NestJS?

**Ideal Answer:**

> NestJS follows MVC naturally: **Model** = Prisma models and services (business logic). **Controller** = NestJS controllers (handle routes and HTTP requests). **View** = The API response (or Handlebars templates if using server-side rendering). In a React + NestJS stack, NestJS handles Model + Controller, and React serves as the View.

---

### Q5: How does MVVM apply to React?

**Ideal Answer:**

> In React: **Model** = API calls, data from backend. **View** = JSX (the rendered output). **ViewModel** = The component logic — useState, useEffect, custom hooks — that manages data, transforms it, and exposes it to the View. When state changes (ViewModel), the JSX (View) automatically re-renders.

---

### Q6: What are the advantages and disadvantages of each?

**MVC:**
- ✅ Simple, well-understood, clear separation
- ✅ Great for server-rendered applications
- ❌ Controller can become bloated ("fat controller")
- ❌ View and Controller can become tightly coupled

**MVVM:**
- ✅ Two-way data binding reduces boilerplate
- ✅ ViewModel is easy to unit test
- ✅ Great for reactive, interactive UIs
- ❌ Data binding debugging can be complex
- ❌ Can be overkill for simple applications

---

### Q7: Can you use MVC and MVVM together?

**Ideal Answer:**

> Yes, and this is exactly what happens in a React + NestJS stack. NestJS (backend) uses MVC — Controllers handle requests, Services contain business logic, and Prisma models represent data. React (frontend) uses MVVM — Components serve as ViewModels that manage state and bind it to JSX Views, while fetching data from the NestJS Model layer via APIs.

# SECTION 12: API DEVELOPMENT

> API questions are critical for full-stack roles. Expect questions about REST principles, HTTP methods, status codes, authentication, and JWT.

---

### Q1: What is a REST API?

**Ideal Answer:**

> REST (Representational State Transfer) is an architectural style for designing APIs. Key principles: (1) Client-server separation. (2) Stateless — each request contains all needed information. (3) Resource-based — URLs represent resources (nouns, not verbs). (4) Use HTTP methods to define actions. (5) Standard response formats (usually JSON).

```
GET    /api/users       → List all users
GET    /api/users/1     → Get user with id 1
POST   /api/users       → Create a new user
PUT    /api/users/1     → Replace user 1 entirely
PATCH  /api/users/1     → Update user 1 partially
DELETE /api/users/1     → Delete user 1
```

---

### Q2: Explain HTTP methods and when to use each.

**Ideal Answer:**

| Method | Purpose | Idempotent | Has Body |
|--------|---------|-----------|----------|
| GET | Retrieve data | Yes | No |
| POST | Create resource | No | Yes |
| PUT | Replace entire resource | Yes | Yes |
| PATCH | Update part of resource | Yes | Yes |
| DELETE | Remove resource | Yes | No |

> **Idempotent** means calling it multiple times produces the same result. POST is NOT idempotent — calling it 3 times creates 3 resources.

---

### Q3: What are HTTP status codes?

**Ideal Answer:**

| Range | Category | Common Codes |
|-------|---------|-------------|
| 2xx | Success | 200 OK, 201 Created, 204 No Content |
| 3xx | Redirection | 301 Moved Permanently, 304 Not Modified |
| 4xx | Client Error | 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 409 Conflict, 422 Unprocessable Entity |
| 5xx | Server Error | 500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable |

> **Key distinction:** 401 = "Who are you?" (not authenticated). 403 = "I know who you are, but you can't do this" (not authorized).

---

### Q4: What is JWT and how does authentication work?

**Ideal Answer:**

> JWT (JSON Web Token) is a compact, self-contained token used for authentication. It has three parts: Header (algorithm), Payload (user data), Signature (verification). The flow:
> 1. User logs in with email/password
> 2. Server verifies credentials
> 3. Server creates a JWT with user info and signs it with a secret key
> 4. Client stores the JWT (localStorage or httpOnly cookie)
> 5. Client sends JWT in the `Authorization: Bearer <token>` header with every request
> 6. Server verifies the token and identifies the user

```typescript
// NestJS Login (simplified)
@Post('login')
async login(@Body() body: LoginDto) {
  const user = await this.authService.validateUser(body.email, body.password);
  if (!user) throw new UnauthorizedException('Invalid credentials');
  
  const token = this.jwtService.sign({ 
    sub: user.id, 
    email: user.email, 
    role: user.role 
  });
  
  return { access_token: token };
}
```

---

### Q5: What is the difference between authentication and authorization?

**Ideal Answer:**

> **Authentication** = Verifying WHO you are (login — proving identity with credentials). **Authorization** = Verifying WHAT you can do (permissions — can this user access this resource?). Authentication comes first, then authorization.

> Example: Logging into the system (authentication). Checking if a regular user can access the admin dashboard (authorization — they can't).

---

### Q6: What is CORS and why does it exist?

**Ideal Answer:**

> CORS (Cross-Origin Resource Sharing) is a browser security mechanism that blocks requests from one domain to another by default. It exists to prevent malicious websites from making requests to your API using a user's session. If your React frontend (localhost:3000) calls your NestJS API (localhost:4000), you need to configure CORS on the API to allow it.

---

### Q7: What is the difference between PUT and PATCH?

**Ideal Answer:**

> PUT replaces the ENTIRE resource — you must send ALL fields. PATCH updates only the fields you send. Example: Updating just a user's name — PUT requires sending name, email, age, etc. PATCH only requires sending the name.

---

### Q8: How do you handle API versioning?

**Ideal Answer:**

> Common approaches: (1) **URL versioning**: `/api/v1/users`, `/api/v2/users`. (2) **Header versioning**: `Accept: application/vnd.myapi.v1+json`. (3) **Query parameter**: `/api/users?version=1`. URL versioning is the most common and explicit.

---

### Q9: What is rate limiting and why is it important?

**Ideal Answer:**

> Rate limiting restricts how many requests a client can make in a time period (e.g., 100 requests per minute). It prevents abuse, DDoS attacks, brute-force login attempts, and protects server resources. Return HTTP 429 (Too Many Requests) when the limit is exceeded.

# SECTION 13: DEBUGGING & TROUBLESHOOTING

> These scenario-based questions test your problem-solving approach. Interviewers want to see your systematic thinking, not just the answer.

---

### Scenario 1: "The website is slow. How would you troubleshoot?"

**Ideal Approach:**

> 1. **Identify WHERE it's slow** — Open browser DevTools → Network tab. Check which requests are slow.
> 2. **Frontend checks:**
>    - Check for large unoptimized images (compress them, use WebP)
>    - Check for unused CSS/JS (bundle analysis)
>    - Check if assets are minified
>    - Check for render-blocking scripts (use `defer` or `async`)
> 3. **API checks:**
>    - Check API response times in Network tab
>    - Look for N+1 query problems (too many database queries)
>    - Check if pagination is implemented (loading 10,000 records at once?)
> 4. **Database checks:**
>    - Run `EXPLAIN ANALYZE` on slow queries
>    - Check for missing indexes on frequently queried columns
>    - Check for full table scans
> 5. **Quick wins:**
>    - Add database indexes
>    - Implement caching (Redis)
>    - Implement pagination
>    - Lazy load images and components

---

### Scenario 2: "API returns 500 Internal Server Error. How do you debug?"

**Ideal Approach:**

> 1. **Check server logs** — The 500 error means something crashed on the server. The log file will have the stack trace with the exact error and line number.
> 2. **Reproduce the error** — Use Postman or curl with the same request. Check the request body, headers, and params.
> 3. **Common causes:**
>    - Unhandled null/undefined (trying to access property of null)
>    - Database connection failure
>    - Missing environment variables
>    - Invalid data types (string where number expected)
>    - Prisma schema mismatch (forgot to run migration)
> 4. **Fix approach:**
>    - Add try/catch blocks around database operations
>    - Validate input with DTOs
>    - Check database connection
>    - Run `npx prisma migrate dev` if schema changed

---

### Scenario 3: "Database query is slow. How do you optimize it?"

**Ideal Approach:**

> 1. **Identify the slow query** — Use `EXPLAIN ANALYZE` to see the query plan.
> 2. **Check for:**
>    - Sequential scan (Seq Scan) instead of Index Scan → Add an index
>    - No WHERE clause → Add filtering
>    - SELECT * → Select only needed columns
>    - N+1 problem → Use JOIN or Prisma `include`
>    - No pagination → Add LIMIT/OFFSET
> 3. **Solutions:**
```sql
-- Before: full table scan
SELECT * FROM orders WHERE user_id = 42;

-- Add index
CREATE INDEX idx_orders_user_id ON orders(user_id);

-- After: uses index, much faster
EXPLAIN ANALYZE SELECT * FROM orders WHERE user_id = 42;
```

---

### Scenario 4: "Login suddenly stopped working. How do you troubleshoot?"

**Ideal Approach:**

> 1. **Check if it's ALL users or specific users** — Try logging in with a test account.
> 2. **Check the browser** — DevTools → Network tab → Check the login request's response code and body.
> 3. **If 401/403:**
>    - Check if the password comparison is working (password_verify in PHP, bcrypt.compare in Node)
>    - Check if the JWT secret key changed
>    - Check if token expiration is too short
> 4. **If 500:**
>    - Check database connection (did the DB go down?)
>    - Check server logs for the stack trace
> 5. **If request never reaches server:**
>    - Check CORS configuration
>    - Check if the API URL changed
>    - Check network connectivity

---

### Scenario 5: "React page is not updating when data changes."

**Ideal Approach:**

> 1. **State mutation** — Most common cause. Mutating state directly instead of creating new references.
>    ```jsx
>    // ❌ Mutating array
>    users.push(newUser);
>    setUsers(users);  // Same reference, React won't re-render
>    
>    // ✅ Creating new array
>    setUsers([...users, newUser]);
>    ```
> 2. **Missing dependency** in useEffect — The effect runs once but doesn't re-run when data changes.
> 3. **Wrong state structure** — Updating the wrong state variable.
> 4. **Component not re-rendering** — Check if parent passes new props. Check if using React.memo incorrectly.

---

### Scenario 6: "NestJS endpoint failing with no clear error."

**Ideal Approach:**

> 1. **Enable detailed error messages** — Make sure your exception filter returns the error message, not just a generic 500.
> 2. **Add logging middleware** — Log every incoming request with method, URL, body, and response status.
> 3. **Check middleware/guard/pipe order** — The NestJS lifecycle is: Middleware → Guards → Interceptors → Pipes → Handler. Any step could be failing silently.
> 4. **Check if pipes are validating correctly** — Missing `ValidationPipe` or DTO validation could cause unexpected behavior.
> 5. **Check Prisma client** — Is it connected? Did you run migrations? Is the model correct?

```sql
CREATE TABLE posts (
  id SERIAL PRIMARY KEY,
  title VARCHAR(255) NOT NULL,
  content TEXT,
  published BOOLEAN DEFAULT FALSE,
  author_id INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE comments (
  id SERIAL PRIMARY KEY,
  content TEXT NOT NULL,
  post_id INTEGER NOT NULL REFERENCES posts(id) ON DELETE CASCADE,
  user_id INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_posts_author ON posts(author_id);
CREATE INDEX idx_comments_post ON comments(post_id);
CREATE INDEX idx_comments_user ON comments(user_id);
```

## Debugging Questions (10)

**Q1:** Why does this code always show the last item's value?
```javascript
var buttons = document.querySelectorAll('.btn');
for (var i = 0; i < buttons.length; i++) {
  buttons[i].addEventListener('click', function() {
    console.log(i);
  });
}
```
**Answer:** `var` is function-scoped. By the time any button is clicked, the loop has finished and `i` equals `buttons.length`. Fix: Use `let` instead of `var`.

---

**Q2:** Why does this React component render infinitely?
```jsx
function App() {
  const [data, setData] = useState([]);
  useEffect(() => {
    fetch('/api/data').then(r => r.json()).then(setData);
  }); // Missing dependency array!
  return <div>{data.length}</div>;
}
```
**Answer:** Missing dependency array `[]` in useEffect. Without it, the effect runs on every render. Setting state causes a re-render, which triggers the effect again → infinite loop. Fix: Add `[]`.

---

**Q3:** Why does this PHP code give "headers already sent" error?
```php
<html>
<body>
<?php
  header('Location: dashboard.php');
?>
```
**Answer:** `header()` must be called before ANY output (including HTML, whitespace, or `echo`). The `<html><body>` is sent before the `header()` call. Fix: Move the PHP code to the top of the file before any HTML.

---

**Q4:** Why does this SQL query return no results even though there is data?
```sql
SELECT * FROM users WHERE name = NULL;
```
**Answer:** You can't compare to NULL with `=`. NULL represents an unknown value. Use `IS NULL` instead: `WHERE name IS NULL`.

---

**Q5:** Why does this button click handler not work in React?
```jsx
<button onClick={handleDelete(item.id)}>Delete</button>
```
**Answer:** `handleDelete(item.id)` is being CALLED immediately during render, not passed as a callback. Fix: Wrap in an arrow function: `onClick={() => handleDelete(item.id)}`.

---

**Q6:** Why does this CSS not center the div vertically?
```css
.container { margin: 0 auto; }
```
**Answer:** `margin: 0 auto` only centers horizontally. For vertical centering, you need the parent to have a defined height and use Flexbox: `display: flex; justify-content: center; align-items: center; height: 100vh;`.

---

**Q7:** Why does this Prisma query return an error?
```typescript
const user = await prisma.user.findUnique({ where: { name: "Juan" } });
```
**Answer:** `findUnique` can only query by fields marked as `@unique` or `@id` in the schema. If `name` isn't unique, use `findFirst` instead.

---

**Q8:** Why does this NestJS endpoint return 400 when the data looks correct?
```typescript
@Post()
create(@Body() dto: CreateUserDto) { ... }
// Sent: { "name": "Juan", "email": "j@e.com", "extra_field": "test" }
```
**Answer:** If `ValidationPipe` is configured with `forbidNonWhitelisted: true`, it rejects requests with properties not in the DTO. The `extra_field` is not defined in `CreateUserDto`. Remove the extra field or set `forbidNonWhitelisted: false`.

---

**Q9:** Why does this code log "undefined"?
```javascript
const user = { name: "Juan", address: { city: "Manila" } };
const { city } = user;
console.log(city);
```
**Answer:** Destructuring `{ city }` looks for `user.city`, which doesn't exist. City is nested inside address. Fix: `const { address: { city } } = user;` or `const city = user.address.city;`.

---

**Q10:** Why does this PHP login check always fail?
```php
if ($password === $stored_hash) {
    echo "Login success";
}
```
**Answer:** You can't compare a plain text password directly with a bcrypt hash — they'll never match. Use `password_verify($password, $stored_hash)` which knows how to compare a plain password against its hashed version.

# SECTION 16: HIDDEN INTERVIEWER QUESTIONS

> These are the tricky follow-up questions senior developers use to test whether you truly understand concepts vs. memorized answers. They'll ask these when your first answer sounds too rehearsed.

---

### After you explain closures:
**"If closures retain references to outer variables, how does that affect memory? Can closures cause memory leaks?"**

**Answer you need:** Yes, closures can cause memory leaks if they hold references to large objects or DOM elements that should have been garbage collected. For example, event listeners in a React component that aren't cleaned up in the useEffect cleanup function will retain references. The fix is to always clean up: remove event listeners, clear timers, and cancel fetch requests in useEffect's return function.

---

### After you explain `useState`:
**"Why can't you call `useState` inside a loop or condition?"**

**Answer you need:** React relies on the ORDER of hook calls to maintain state between renders. If you put a hook inside a condition, it might not run on every render, shifting the order and causing React to return the wrong state values. This is the "Rules of Hooks" — hooks must always be called in the same order at the top level of the component.

---

### After you explain JWT:
**"If JWTs are self-contained, how do you invalidate a token when a user logs out?"**

**Answer you need:** This is a real weakness of JWTs. Since they're stateless, you can't truly "invalidate" a token before it expires. Common solutions: (1) Short expiration times (15 minutes) with refresh tokens. (2) A server-side token blacklist (defeats the stateless purpose but sometimes necessary). (3) Change the JWT secret (invalidates ALL tokens). (4) Store a "token version" per user and check it on every request.

---

### After you explain database indexing:
**"If indexes speed up reads, why not index every column?"**

**Answer you need:** Because indexes slow down writes (INSERT, UPDATE, DELETE). Every time you modify data, all relevant indexes must be updated too. Indexes also consume disk space. The trade-off is: index columns frequently used in WHERE, JOIN, and ORDER BY clauses, but don't index columns rarely queried or frequently updated.

---

### After you explain React Virtual DOM:
**"Is the Virtual DOM always faster than direct DOM manipulation?"**

**Answer you need:** No. For simple, one-off DOM updates (like changing a single element's text), direct DOM manipulation is faster because there's no diffing overhead. The Virtual DOM's advantage is in complex UIs with many updates — it batches changes and minimizes actual DOM operations. For very performance-critical animations, developers sometimes use `useRef` to manipulate the DOM directly.

---

### After you explain prepared statements:
**"Can prepared statements prevent ALL types of SQL injection?"**

**Answer you need:** They prevent injection in data values, but they can't parameterize table names, column names, or ORDER BY directions. For dynamic table/column names, you need a whitelist approach — validate against a list of allowed values before inserting into the query string.

---

### After you explain REST:
**"What happens when REST doesn't fit your use case well?"**

**Answer you need:** REST can be inefficient when you need: (1) Real-time updates → Use WebSockets or Server-Sent Events. (2) Complex queries where the client needs different data shapes → Consider GraphQL. (3) Sending/receiving large file streams → Use gRPC or streaming APIs. REST is great for standard CRUD, but it's not the only option.

---

### After you explain `===`:
**"How does JavaScript compare objects with `===`?"**

**Answer you need:** `===` compares objects by REFERENCE, not by value. Two objects with identical contents are NOT equal unless they are literally the same object in memory. `{a: 1} === {a: 1}` is `false`. This is why React state updates require new object/array references to trigger re-renders.

---

### After you explain Prisma:
**"What are the downsides of using an ORM like Prisma?"**

**Answer you need:** (1) Performance overhead — ORM-generated queries may not be as optimized as hand-written SQL, especially for complex queries. (2) Learning curve — you need to learn the ORM's API in addition to SQL. (3) Abstraction leaks — for complex queries (recursive CTEs, advanced window functions), you end up writing raw SQL anyway. (4) Migration challenges — ORMs manage your schema, which can conflict with DBAs or complex migration scenarios.

---

### After you explain dependency injection:
**"Why is dependency injection important for testing?"**

**Answer you need:** DI lets you replace real dependencies with mock/fake implementations during tests. Instead of your service using the real database, you inject a mock PrismaService that returns fake data. This makes tests fast (no real database), isolated (tests don't affect each other), and predictable (controlled data).

---

### After you explain MVVM:
**"In React, can you point to exactly where the 'ViewModel' is?"**

**Answer you need:** In React, the ViewModel isn't a separate file — it's the component's logic itself. The `useState`, `useEffect`, event handlers, and data transformation code within the component act as the ViewModel. Custom hooks are the closest thing to a formal ViewModel extraction — they encapsulate the logic separately from the View (JSX).

---

### After you explain error handling:
**"What's the difference between an error and an exception?"**

**Answer you need:** An error is a broader term for anything that goes wrong. An exception is a specific type of error that can be caught and handled programmatically (using try/catch). Not all errors are exceptions — syntax errors, out-of-memory errors, and hardware failures can't be caught with try/catch. In practice, you handle expected exceptions (not found, validation, network) and let unexpected errors crash with proper logging.

# SECTION 17: INTERVIEW STRATEGY GUIDE

> This section covers HOW to answer, not just WHAT to answer. These strategies are especially effective for Philippine IT company interviews.

---

## How to Answer Confidently

1. **Pause before answering.** Take 2-3 seconds to organize your thoughts. Don't rush — a structured answer delivered calmly beats a fast, rambling one.

2. **Use the "Answer First, Explain Second" approach.** State the direct answer, THEN elaborate.
   - ❌ "So, um, basically there are different types of, like, variables in JavaScript and they have different scoping..."
   - ✅ "There are three: var, let, and const. The key difference is scope — var is function-scoped while let and const are block-scoped."

3. **Speak in terms the interviewer uses.** Mirror their terminology. If they say "endpoints," don't switch to "routes."

4. **It's okay to think out loud.** "That's a great question. Let me think about this..." shows confidence, not weakness.

---

## How to Structure Answers

**For concept questions, use this formula:**
> Definition → Key difference/detail → Example → Real-world analogy (optional)

**For coding questions, use this approach:**
1. Restate the problem to confirm understanding
2. Discuss your approach before coding
3. Write the code
4. Walk through it with a test case
5. Mention edge cases

**For "tell me about a time" questions, use STAR:**
> **S**ituation → **T**ask → **A**ction → **R**esult

---

## What Interviewers Are Actually Looking For

| What they ask | What they're really testing |
|--------------|---------------------------|
| "Tell me about yourself" | Communication skills, confidence, motivation |
| "Explain [concept]" | Can you teach it? Do you truly understand? |
| "Write code for..." | Problem-solving approach, code quality, edge cases |

## What to Do When You Don't Know the Answer

**Never say:** "I don't know." (and stop talking)

**Instead, use one of these:**

1. **"I'm not sure about the exact details, but here's what I do know..."** → Share related knowledge.

2. **"I haven't used that specific feature yet, but based on my understanding of [related concept], I think..."** → Show reasoning ability.

3. **"That's a great question. I haven't encountered that in my projects yet, but I'd approach learning it by [reading docs / building a small project / checking the official documentation]."** → Show learning methodology.

4. **"I don't want to give you inaccurate information, but I believe [educated guess]. I would verify this by [how you'd research it]."** → Show intellectual honesty.

## Common Internship Interview Mistakes

1. **Memorizing answers without understanding.** Interviewers will ask follow-ups. If you can't explain WHY, the memorized answer hurts more than it helps.

2. **Not having a project to talk about.** Even a simple CRUD app is better than nothing. Build something before the interview.

3. **Saying "I used a tutorial" without modifications.** It's fine to follow tutorials, but say "I followed a tutorial as a starting point, then I customized it by adding [feature]."

4. **Speaking too softly or too fast.** Practice answering in front of a mirror or with a friend. Record yourself.

5. **Not asking questions.** This signals disinterest. Always have 2-3 prepared questions.

6. **Apologizing for being a beginner.** Don't say "I'm sorry, I'm just a student." Own your level: "I'm currently at an intermediate level and actively learning."

7. **Lying about skills.** If you list "NestJS" on your resume, they WILL ask about it. Only list technologies you can discuss at a basic level.

8. **Ignoring the soft skills portion.** In Philippine companies especially, cultural fit and "magaling makisama" (gets along well with others) matters as much as technical ability.

## How to Stand Out as a First-Time Intern

1. **Show your GitHub.** Even with simple projects, an active GitHub shows initiative. Clean READMEs, organized code, and commit history make a big impression.

2. **Build a portfolio site.** Even a one-page site with your projects shows you can ship something.

7. **Practice with a friend.** Do a mock interview. Have them ask random questions from this reviewer. The more you practice speaking your answers aloud, the more natural they'll sound.
